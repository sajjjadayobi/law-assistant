### Harness Engineering For Long Horizon Real Tasks: End of Proxies
Agents still face challenges working across many context windows. Longer-horizon problems are harder for AI to solve completely. Imagine a software project staffed by engineers working in shifts, where each new engineer arrives with no memory of what happened on the previous shift. For long running agents we have compaction in Claude Code and Agent SDK. However, compaction isn’t sufficient. We want incremental progress across many context windows. The new benchmarks measure models’ abilities to do economically useful work. Tasks that actually matter.
- Evals: We don't need general proxy task benchmarks we want results in real environments!
Why Long Horizon Real Tasks? These goal-oriented AI systems need more than predictive or generative capability. Humans have the extraordinary capability of consolidating diverse information collected over extended periods of time into a consistent world-view that drives execution of complex goals in evolving environments. Shifting focus to these long-horizon tasks and goal-oriented AI systems has two major benefits. They require persistent memory, ability to focus on a goal over a long time horizon, responsiveness to real-time human feedback, and the ability to cope with uncertainty in an evolving environment.  They also require effective interface.
- We must forget proxy tasks and only focus on end goal real tasks. Semantic Parsing Lessons.
Problem: even a frontier coding model running on the Claude Agent SDK in a loop across multiple context windows will fall short of building a production-quality web app if it’s only given a high-level prompt, such as “build a clone of claude.ai.”
Failures: the agent tended to try to do too much at once—essentially to attempt to one-shot the app. Often, this led to the model running out of context in the middle of its implementation, leaving the next session to start with a feature half-implemented and undocumented. After some features had already been built, a later agent instance would look around, see that progress had been made, and declare the job done.
#### Solution: multi-context window workflows
Initializer: The very first agent session uses a prompt that asks the model to set up the initial environment: an init.sh script, a progress.txt  file that keeps a log of what agents have done, and an initial git commit that shows what files were added. We prompted the initializer agent to write a comprehensive file of feature list expanding on the user’s initial prompt.
Feature Creator: The agent was then asked to work on only one feature at a time. This incremental approach turned out to be critical to addressing the agent’s tendency to do too much at once. Ask the model to commit its progress to git with descriptive commit messages and to write summaries of its progress in a progress file. It runs tests and make the state of the repo to be able to merge to a main branch.
Testing: In the case of building a web app, Claude mostly did well at verifying features end-to-end once explicitly prompted to use browser automation tools and do all testing as a human user would.

### Anthropic on Multi Agents: It's still Context Engineering
A multi-agent system consists of multiple agents (LLMs autonomously using tools in a loop) working together. Our Research feature involves an agent that plans a research process based on user queries, and then uses tools to create parallel agents that search for information simultaneously. 
When people conduct research, they tend to continuously update their approach based on discoveries, following leads that emerge during investigation.
- When a task is like this it's better to go more agentic instead of fixed workflows. 
Each subagent also provides separation of concerns—distinct tools, prompts, and exploration trajectories—which reduces path dependency and enables thorough, independent investigations.
We found that a multi-agent system with Claude Opus 4 as the lead agent and Claude Sonnet 4 subagents outperformed single-agent Claude Opus 4 by 90.2% on our internal research eval. We found that token usage by itself explains 80% of the variance, with the number of tool calls and the model choice as the two other explanatory factors. In our data, agents typically use about 4× more tokens than chat interactions, and multi-agent systems use about 15× more tokens than chats. Multi-agent systems require tasks where the value of the task is high enough to pay for the increased performance. 
We’ve found that multi-agent systems excel at valuable tasks that involve heavy parallelization, information that exceeds single context windows, and interfacing with numerous complex tools.
#### On Planing
planning should be decoupled from execution: You ask the agent to first generate a plan, and only after this plan is validated is it executed. The plan can be validated using heuristics. A plan can also be validated using AI judges. To speed up the process, instead of generating plans sequentially, you can generate several plans in parallel and ask the evaluator to pick the most promising one.
- You must understand the the user intent, if it's irrelevant you reject the query or clarify it.
Hierarchical Planing: There’s a planning/execution tradeoff. A detailed plan is harder to generate, but easier to execute. A higher-level plan is easier to generate, but harder to execute. An approach to circumvent this tradeoff is to plan hierarchically. Use the best model possible for planning and a faster model for sub-task execution.
- Progress monitoring: Have a plan with checkmarks an update on each call and show it.
#### Context Engineering of Multi Agent Systems
To iterate on prompts, you must understand their effects. To help us do this, we built simulations, then watched agents work step-by-step. This immediately revealed failure modes with error analysis. Effective prompting relies on developing an accurate mental model of the agent, which can make the most impactful changes obvious.
Each subagent needs an objective, an output format, guidance on the tools and sources to use, and clear task boundaries. Without detailed task descriptions, agents duplicate work, leave gaps, or fail to find necessary information.
Agents struggle to judge appropriate effort for different tasks, so we embedded scaling rules in the prompts. Simple fact-finding requires just 1 agent with 3-10 tool calls, direct comparisons might need 2-4 subagents with 10-15 calls each, and complex research might use more than 10 subagents with clearly divided responsibilities. These explicit guidelines help the lead agent allocate resources efficiently and prevent over investment in simple queries, which was a common failure mode in our early versions.
Tool design and selection are critical. Agent-tool interfaces are as critical as human-computer interfaces. Using the right tool is efficient—often, it’s strictly necessary. For instance, an agent searching the web for context that only exists in Slack is doomed from the start. With MCP servers that give the model access to external tools, this problem compounds, as agents encounter unseen tools with descriptions of wildly varying quality. We gave our agents explicit heuristics: for example, examine all available tools first, match tool usage to user intent, search the web for broad external exploration, or prefer specialized tools over generic ones.
Takeaway: Our prompting strategy focuses on instilling good heuristics rather than rigid rules. We studied how skilled humans approach research tasks and encoded these strategies in our prompts—strategies like decomposing difficult questions into smaller tasks, carefully evaluating the quality of sources, adjusting search approaches based on new information, and recognizing when to focus on depth vs. breadth. We also proactively mitigated unintended side effects by setting explicit guardrails to prevent the agents from spiraling out of control.
We implemented patterns where agents summarize completed work phases and store essential information in external memory before proceeding to new tasks for better context management.
#### Evaluation of Multi Agent Systems
Traditional evaluations often assume that the AI follows the same steps each time: given input X, the system should follow path Y to produce output Z. Because we don’t always know what the right steps are, we usually can't just check if agents followed the “correct” steps we prescribed in advance. Instead, we need flexible evaluation methods that judge whether agents achieved the right outcomes while also following a reasonable process.
Start evaluating immediately with small samples. In early agent development, changes tend to have dramatic impacts because there is abundant low-hanging fruit. it’s best to start with small-scale testing right away with a few examples, rather than delaying until you can build more thorough evals. 
- We found success focusing on end-state evaluation rather than turn-by-turn analysis.
LLM-as-judge evaluation scales when done well. We experimented with multiple judges to evaluate each component, but found that a single LLM call with a single prompt outputting scores from 0.0-1.0 and a pass-fail grade was the most consistent and aligned with human judgements. Using an LLM as a judge allowed us to scalably evaluate hundreds of outputs. As we more more into human error analysis followed by agentic context engineering we must focus more on the E2E evals that focus on the product goal of the agent with detailed feedback. these detailed comments must for agentic context engineering plus dynamic few shot examples for the judge.
- You first start by eval examples that have clear correct answers with reference answers.
Human evaluation catches what automation misses. People testing agents find edge cases that evals miss. These include hallucinated answers on unusual queries, system failures, or subtle source selection biases. Even in a world of automated evaluations, manual testing remains essential. Ask people to write free form feedback about why the agent fail and how to fix it.
Agentic systems need mental models not instructions: The best prompts for these agents are not just strict instructions, but frameworks for collaboration that define the division of labor, problem-solving approaches, and effort budgets. Getting this right relies on careful prompting and tool design, solid heuristics, observability, and tight feedback loops.
#### Engineering and Deployment of Multi Agent Systems
Agents are stateful and errors compound. Agents can run for long periods of time, maintaining state across many tool calls. Without effective mitigations, minor system failures can be catastrophic for agents. When errors occur, we can't just restart from the beginning: restarts are expensive and frustrating for users. Instead, we built systems that can resume from where the agent was when the errors occurred. We also use the model’s intelligence to handle issues gracefully: for instance, letting the agent know when a tool is failing and letting it adapt works surprisingly well. We combine the adaptability of AI agents with deterministic safeguards like retry logic and regular checkpoints.
We need new types and meta observability with agents themselves. Beyond basic logging of traces we log number of tool calls and compute summary stats for agent decisions with agents themselves to monitor traces and logs for new patterns, error analysis, annotations, finding bugs and finding good few shot examples and anomalies to check later. 
- You need full production tracing to create a real feedback loop with agentic observability.

### Bitter Lesson in Building Agents: Building the Right Abstraction
To build agents that improve with better underlying models and scale with more compute, money, and intelligence, consider the following principles. Focus on flexible architectures that can adapt as underlying models and available data improve, rather than highly specialized designs that might become obsolete.
E.g: Give the model tools especially python code interpreter, ask to run the request task based on the input file and generate 50 answers. Use a verifier to choose the best answer based on unit tests. It can be under parallel test time compute increasing.
Model Agnosticism (where possible): Design agent architectures that are not overly tied to a specific underlying model. This allows for easier adoption of newer, more powerful models as they become available.
Continuous Learning and Adaptation: Implement mechanisms for agents to continuously learn from new data and experiences to improve their performance over time.
Reflection & Error Recovery: Failing is part of the process as long as you can reflect on that process. planning is not complete without reflective behavior. Reflection generates insights that help uncover errors to be corrected. You can have a single reflection agent that criticize and check the plan and the result of each execution steps or have many reflection agent for each step.
Monitoring and Observability: Build in comprehensive logging, monitoring, and observability from the start to track agent behavior, identify issues, and understand performance. "Silent failures" can be particularly problematic. Understanding why an agent makes certain decisions can be difficult, hindering debugging and trust.
Research Phase: Before acting, agents must verify they understand the request and it has correct assumptions. Ask and Explore the Environment. Failing fast with questions is more reliable than confidently doing the wrong thing. Users forgive questions, not mistakes.
Human-in-the-Loop: A great agent is one that can say I don't know and ask for help in the right UX. It can ask good questions at the right step in the process to get the job done fast and great.
Start with Capable Models: Use the best available models, especially during prototyping and development. This allows you to see the potential of an LLM-driven approach and reduces the temptation to over-engineer solutions to compensate for less capable models.
Anticipate Advancements: While impossible to predict the future perfectly, design with flexibility in mind, expecting that models will become more capable and data will become more abundant.
Iterate and Delete Code: As underlying models become more intelligent, be prepared to delete the scaffolding and custom code you previously wrote to work around their limitations. This is a sign that you're aligning with the "bitter lesson."
Design End to End Evals: with your agent design chaining so fast, first focus on the end goal of the agent then with error analysis and data observibility go to some components. Understanding how well your agent performs, where it fails, and how its performance changes as underlying models evolve is key. This includes evaluating aspects like the reliability of tool usage and the number of steps an agent takes to complete a task (convergence).
Parallelization: Design agents and workflows that can be parallelized to take advantage of distributed computing resources. Keep in mind that you can use more test time compute sequentially or parallel.

Building Trust in your Product: Humble Transparent Agents
Trust is unfortunately multidimensional, which makes it difficult to build and easy to lose.
First, know what you don’t know: The best feature of your product is to tell I don't know with confidence. we carefully analyze each data source we think might be relevant to a question; if we think that none of them actually help solve the problem, we never give ourselves a chance to produce a misleading answer. That confidence in “I don’t know” answers has meant that many of our customers use those answers as signposts for areas that need to be better documented.
- Your I don't know feedback loop for better knowledge and lack of context.
- Correctly give the user next steps when you fail and ask for right information and context
Second, don’t make the same mistake twice: if you make the same mistake repeatedly, the feedback that your customers give you seems to be wasteful. When you correct Devin or ask it to make a stylistic change to the code it generated, it automatically identifies that as general guidance and adds that guidance to a shared knowledge base. When it uses that shared knowledge in a future piece of work, it notifies you so you knew what principles it was relying on and were aware that it was steadily getting better over time.
- Make sure the agent understand and apply the feedback and adds that to it's knowledge base.
