# Database Connection Layer - Plan

## What I'm Building

A SQLAlchemy-based database connection layer for PostgreSQL that provides:
- Singleton database engine with connection pooling
- Session management (thread-local sessions for requests)
- ORM models for `documents` and `relations` tables
- Core database utilities (get_session(), get_connection(), async session support)
- Type-safe database operations with Pydantic integration

This layer will be the foundation for all database queries in the Law Agent, providing clean abstractions over raw SQL while maintaining full-text search capabilities and relationship traversal.

## Why This Matters

The Law Agent depends entirely on PostgreSQL for document storage and search. Without a proper connection layer:
- Every module would handle connections independently → connection leaks and resource exhaustion
- No centralized session management → thread safety issues in multi-threaded environments
- ORM models are scattered or missing → inconsistent data access patterns
- Testing requires manual connection mocking → harder to test

A well-designed connection layer provides:
- **Resource efficiency**: Connection pooling reuses connections instead of creating new ones
- **Thread safety**: Thread-local sessions prevent cross-request data leaks
- **Type safety**: SQLAlchemy ORM with Pydantic validation ensures correct data types
- **Testability**: Easy to mock or use test database with dependency injection
- **Consistency**: Single source of truth for database operations

## Key Design Decisions

### Decision 1: SQLAlchemy 2.0+ ORM vs Raw SQL
**Decision**: Use SQLAlchemy 2.0+ ORM with declarative base for type-safe queries
**Why**:
- Type hints for all queries (IDE autocomplete, mypy validation)
- Automatic relationship loading (can follow document relations with `.relations`)
- Migration-ready (future alembic support)
- Clean Python API (fewer raw SQL strings to debug)
- Pydantic integration for validation

**Alternatives Considered**:
- Raw psycopg2: Faster but loses type safety, harder to maintain
- SQLQuery builder (SQLAlchemy core): Middle ground but less convenient

**Trade-offs**: Slight performance overhead vs major gains in maintainability and type safety

### Decision 2: Singleton Engine Pattern with Connection Pool
**Decision**: Single SQLAlchemy engine per application with QueuePool for connection pooling
**Why**:
- Applications typically need one engine (not one per request)
- QueuePool (SQLAlchemy default) manages pool size, recycling, and connection overflow gracefully
- Configured from settings: min_size, max_size, max_overflow
- Thread-safe by design

**Alternatives Considered**:
- New engine per request: Wasteful, slower connections
- No pooling: Connection exhaustion under load

**Trade-offs**: Pool configuration needs tuning per deployment, but standard defaults work

### Decision 3: Thread-Local Sessions with Context Manager
**Decision**: Use `scoped_session` for thread-local session management + context managers for explicit cleanup
**Why**:
- Each thread gets its own session (no cross-request contamination)
- Automatic cleanup with `with get_session()` pattern
- Works with both sync and async code paths (initially sync only)
- Safe for multi-threaded Chainlit UI

**Alternatives Considered**:
- Global session: Cross-request data leaks, threading issues
- Manual session creation: Boilerplate, easy to forget cleanup
- AsyncSession: Not needed yet (agent is sync), adds complexity

**Trade-offs**: Slightly more code, but thread safety is critical

### Decision 4: Two ORM Models (Document, Relation) vs Denormalization
**Decision**: Create clean SQLAlchemy ORM models matching the migrated database schema
**Why**:
- `documents` table: One row per document with full_content, summary, tags, search_vector
- `relations` table: Directed edges with src_doc_id, dst_doc_id, relation_type
- Relationship properties on Document model: `Document.relations_to` and `Document.relations_from` for traversal

**Alternatives Considered**:
- Denormalize all relations into document: Harder to query, wastes space
- No relationships defined: Have to join manually every time

**Trade-offs**: Clear models, slight JOIN overhead vs much cleaner queries

### Decision 5: Full-Text Search Support
**Decision**: Map PostgreSQL `tsvector` search_vector column but use raw SQL for FTS queries (SQLAlchemy FTS support is limited)
**Why**:
- `search_vector` is auto-generated by PostgreSQL triggers
- FTS queries need tsquery which SQLAlchemy doesn't model well
- Raw SQL for FTS is clear and performant
- Regular ORM queries for everything else (get_document, relations, etc.)

**Alternatives Considered**:
- Custom SQLAlchemy TypeDecorator for tsvector: Over-engineered
- All raw SQL: Lose ORM benefits for regular queries

**Trade-offs**: Mix of ORM + raw SQL, but justified for FTS performance

## Open Questions

- [ ] **Question 1: Async Support Now or Later?**
  - Current status: Agent is synchronous (no async)
  - Decision: Start with sync-only now (simpler), add AsyncSession in Phase 4 when needed for concurrent operations
  - Rationale: YAGNI - add when actually needed, not speculatively

- [ ] **Question 2: Relationship Eager/Lazy Loading?**
  - SQLAlchemy has eager loading (joinedload) vs lazy loading (default, loads on access)
  - Decision: Use lazy loading by default for simplicity, optimize with eager loading later if needed
  - Rationale: Simpler queries now, profiling will tell us where optimization is needed

- [ ] **Question 3: Custom Query Session or Dependency Injection?**
  - How should tools (search_documents, get_document) get database access?
  - Decision: Use dependency injection via PydanticAI tool context, or pass session as argument
  - Action: Defer this to when tools are implemented (Task 3.x)

- [ ] **Question 4: Schema Alembic Migrations Now or Later?**
  - Should we set up alembic for future schema changes?
  - Decision: Not now. Database is fixed (migrated in Phase 1), no expected schema changes.
  - Rationale: YAGNI - add when we need to modify schema
